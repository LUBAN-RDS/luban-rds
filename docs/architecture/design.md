# 设计决策

本部分详细介绍了 Luban-RDS 的重要设计决策，包括技术选型、架构设计和实现细节的理由和权衡。

## 1. 技术选型

### 1.1 Java 语言

**决策**：选择 Java 作为开发语言

**理由**：
- **跨平台**：Java 代码可以在任何支持 JVM 的平台上运行
- **生态成熟**：丰富的库和工具，如 Netty、LuaJ 等
- **性能优异**：JVM 优化成熟，性能稳定
- **并发支持**：内置的并发库和线程管理
- **易于维护**：面向对象设计，代码结构清晰

**权衡**：
- **启动时间**：相比 C 语言实现的 Redis，启动时间较长
- **内存占用**：JVM 本身需要一定的内存开销
- **GC 暂停**：垃圾收集可能导致短暂的服务暂停

### 1.2 Netty 框架

**决策**：选择 Netty 作为网络框架

**理由**：
- **高性能**：基于 NIO 的非阻塞 IO，支持高并发
- **成熟稳定**：广泛应用于生产环境，如 Elasticsearch、Kafka 等
- **易于使用**：提供了丰富的网络编程抽象
- **可扩展**：模块化设计，易于扩展和定制
- **安全可靠**：内置了多种安全特性

**权衡**：
- **学习曲线**：相比简单的 Socket 编程，学习成本较高
- **配置复杂**：需要配置线程池、缓冲区等参数
- **依赖管理**：需要管理 Netty 及其依赖

### 1.3 LuaJ 库

**决策**：选择 LuaJ 作为 Lua 脚本执行引擎

**理由**：
- **纯 Java 实现**：与 Java 无缝集成，无需本地库
- **性能优异**：经过优化的 Lua 解释器
- **安全可控**：可配置的沙箱环境
- **兼容 Lua 5.2**：支持 Lua 5.2 的语法和标准库
- **活跃维护**：持续更新和改进

**权衡**：
- **性能**：相比原生 Lua 实现，执行速度较慢
- **功能限制**：某些 Lua 扩展库可能不支持
- **内存占用**：每个 Lua 状态需要一定的内存

### 1.4 Maven 构建工具

**决策**：选择 Maven 作为项目管理和构建工具

**理由**：
- **依赖管理**：自动管理项目依赖
- **构建生命周期**：标准化的构建流程
- **多模块支持**：适合构建复杂的多模块项目
- **插件生态**：丰富的插件支持各种构建需求
- **广泛使用**：Java 生态系统中最常用的构建工具

**权衡**：
- **配置复杂**：pom.xml 配置文件可能变得复杂
- **构建速度**：相比 Gradle，构建速度较慢
- **灵活性**：相比 Gradle，脚本编写不够灵活

## 2. 架构设计

### 2.1 分层架构

**决策**：采用分层架构设计

**理由**：
- **职责清晰**：各层有明确的职责边界
- **易于维护**：每层可以独立开发和测试
- **便于扩展**：可以在不影响其他层的情况下扩展某一层
- **代码复用**：各层的组件可以被其他模块复用
- **易于理解**：分层架构符合人们的思维习惯

**层次划分**：
- **网络层**：处理网络连接和数据传输
- **协议层**：解析和编码 Redis 协议
- **命令处理层**：处理 Redis 命令
- **存储层**：管理内存数据
- **持久化层**：处理数据持久化

### 2.2 模块化设计

**决策**：采用模块化设计，将项目分为多个 Maven 模块

**理由**：
- **功能分离**：每个模块专注于特定功能
- **依赖管理**：清晰的模块依赖关系
- **并行开发**：不同模块可以并行开发
- **测试方便**：可以单独测试每个模块
- **部署灵活**：可以根据需要部署不同的模块

**模块划分**：
- **luban-rds-core**：核心业务逻辑
- **luban-rds-protocol**：协议解析
- **luban-rds-server**：网络服务
- **luban-rds-persistence**：持久化
- **luban-rds-client**：Java 客户端
- **luban-rds-common**：公共工具
- **luban-rds-bin**：启动和基准测试
- **luban-rds-spring-boot-starter**：Spring Boot 集成

### 2.3 内存存储设计

**决策**：使用 Java 集合作为内存存储的基础

**理由**：
- **实现简单**：直接使用 Java 内置集合，减少开发复杂度
- **性能可靠**：Java 集合经过优化，性能稳定
- **易于维护**：代码可读性高，便于维护
- **灵活性高**：可以根据需要扩展和定制

**数据结构选择**：
- **String**：直接存储字符串值
- **List**：使用 `LinkedList`，适合频繁的插入和删除操作
- **Set**：使用 `HashSet`，提供 O(1) 的成员检查
- **Hash**：使用 `HashMap`，适合键值对存储
- **ZSet**：使用跳表结构，支持按分数排序

**权衡**：
- **内存开销**：Java 对象有一定的内存开销
- **序列化**：需要额外的序列化和反序列化开销
- **并发控制**：需要手动处理并发访问

### 2.4 直接操作优化

**决策**：对于集合操作，直接修改底层集合，避免数据复制

**理由**：
- **性能提升**：避免了不必要的数据复制，提高操作速度
- **内存节省**：减少了临时对象的创建，节省内存
- **响应更快**：操作执行时间更短，响应更快

**适用操作**：
- **Hash**：`HSET`、`HDEL` 等操作
- **List**：`LPUSH`、`RPUSH`、`LPOP`、`RPOP` 等操作
- **Set**：`SADD`、`SREM` 等操作
- **ZSet**：`ZADD`、`ZREM` 等操作

**权衡**：
- **线程安全**：需要更复杂的并发控制
- **代码复杂度**：实现难度增加
- **错误处理**：需要处理更多的边界情况

## 3. 功能设计

### 3.1 完全兼容 Redis 协议

**决策**：完全实现 Redis 序列化协议（RESP）

**理由**：
- **客户端兼容**：可以直接使用现有的 Redis 客户端
- **生态集成**：可以与 Redis 生态系统的工具集成
- **学习成本低**：用户可以使用已有的 Redis 知识
- **标准化**：遵循行业标准，便于互操作

**实现**：
- 支持所有 RESP 数据类型：Simple String、Error、Integer、Bulk String、Array
- 支持所有 Redis 命令的协议格式
- 保持与 Redis 相同的错误处理和返回值格式

### 3.2 Lua 脚本支持

**决策**：实现完整的 Lua 脚本支持

**理由**：
- **功能扩展**：用户可以通过 Lua 脚本扩展功能
- **原子操作**：脚本执行期间不会被其他命令中断
- **减少网络往返**：复杂逻辑可以在服务器端执行
- **与 Redis 兼容**：支持 Redis 的 Lua API

**实现**：
- 支持 `EVAL`、`EVALSHA`、`SCRIPT` 命令族
- 提供完整的 Redis Lua API：`redis.call()`、`redis.pcall()`、`redis.error_reply()`、`redis.status_reply()`、`redis.sha1hex()`
- 支持脚本缓存、超时控制、沙箱模式

### 3.3 持久化机制

**决策**：实现 RDB 和 AOF 两种持久化机制

**理由**：
- **数据安全**：防止数据丢失
- **灾难恢复**：在服务重启后恢复数据
- **灵活性**：用户可以根据需要选择合适的持久化策略
- **与 Redis 兼容**：使用与 Redis 相同的持久化格式

**RDB 特点**：
- **优点**：文件紧凑，恢复速度快
- **缺点**：可能丢失最近的数据修改

**AOF 特点**：
- **优点**：数据安全性高，可配置不同的同步策略
- **缺点**：文件较大，恢复速度较慢

### 3.4 发布订阅

**决策**：实现完整的发布订阅功能

**理由**：
- **实时通信**：支持实时消息传递
- **解耦设计**：发布者和订阅者解耦
- **灵活扩展**：可以轻松添加新的订阅者
- **与 Redis 兼容**：支持 Redis 的发布订阅命令

**实现**：
- 支持 `SUBSCRIBE`、`UNSUBSCRIBE`、`PUBLISH`、`PSUBSCRIBE`、`PUNSUBSCRIBE` 命令
- 实现 `PubSubManager` 管理频道和订阅关系
- 支持模式订阅和消息广播

### 3.5 Spring Boot 集成

**决策**：提供 Spring Boot starter 模块

**理由**：
- **易用性**：简化在 Spring Boot 应用中的使用
- **自动配置**：提供合理的默认配置
- **生态集成**：与 Spring 生态系统无缝集成
- **开发效率**：减少配置和样板代码

**实现**：
- 自动配置 Luban-RDS 服务器
- 注册 `RedisTemplate` 为 Spring Bean
- 提供配置属性，支持自定义配置

## 4. 性能设计

### 4.1 并发控制

**决策**：采用细粒度的并发控制策略

**理由**：
- **性能提升**：减少锁竞争，提高并发性能
- **资源利用**：充分利用多核 CPU 资源
- **响应速度**：减少线程等待时间

**实现**：
- **分段锁**：对不同的数据段使用不同的锁
- **读写锁**：区分读操作和写操作，提高并发度
- **无锁数据结构**：对于某些场景，使用无锁数据结构
- **线程池**：使用线程池处理命令执行，避免线程创建开销

### 4.2 内存管理

**决策**：实现高效的内存管理策略

**理由**：
- **资源节约**：减少内存使用，降低成本
- **性能提升**：减少内存分配和垃圾收集开销
- **稳定性**：避免内存泄漏和溢出

**实现**：
- **对象池**：重用频繁创建的对象
- **延迟删除**：采用惰性删除和定期删除相结合的策略
- **内存限制**：支持设置最大内存使用限制
- **淘汰策略**：当内存达到限制时，使用 LRU、LFU 等策略淘汰键

### 4.3 网络优化

**决策**：优化网络处理性能

**理由**：
- **响应速度**：减少网络延迟，提高响应速度
- **吞吐量**：增加单位时间内处理的请求数
- **稳定性**：提高网络连接的稳定性

**实现**：
- **Netty 优化**：配置合适的线程池大小、缓冲区大小等参数
- **连接复用**：支持连接池，减少连接建立开销
- **批量操作**：支持管道（pipeline）批量执行命令
- **压缩传输**：对于大数据，支持压缩传输

### 4.4 持久化优化

**决策**：优化持久化性能

**理由**：
- **减少 IO 开销**：降低磁盘 IO 对服务性能的影响
- **数据安全性**：确保数据可靠持久化
- **存储空间**：减少持久化文件的大小

**实现**：
- **异步持久化**：使用后台线程执行持久化操作
- **批量写入**：批量写入数据，减少 IO 次数
- **压缩存储**：使用压缩格式存储数据
- **AOF 重写**：定期重写 AOF 文件，减小文件大小

## 5. 安全设计

### 5.1 Lua 沙箱

**决策**：实现 Lua 脚本沙箱

**理由**：
- **防止恶意代码**：限制脚本的危险操作
- **资源保护**：防止脚本消耗过多资源
- **安全隔离**：隔离脚本执行环境

**实现**：
- **模块限制**：可配置允许使用的 Lua 模块
- **函数限制**：可配置禁止使用的危险函数
- **内存限制**：限制脚本大小和返回值大小
- **执行时间限制**：防止脚本长时间运行

### 5.2 认证机制

**决策**：实现密码认证机制

**理由**：
- **访问控制**：防止未授权访问
- **安全性**：保护敏感数据
- **与 Redis 兼容**：支持 Redis 的 AUTH 命令

**实现**：
- 支持配置密码
- 实现 AUTH 命令进行认证
- 未认证的连接只能执行 AUTH 命令

### 5.3 命令限制

**决策**：支持限制危险命令的使用

**理由**：
- **防止误操作**：避免用户执行危险命令
- **安全性**：减少安全风险
- **与 Redis 兼容**：支持 Redis 的命令重命名机制

**实现**：
- 支持通过配置禁用或重命名危险命令
- 常见的危险命令：`FLUSHALL`、`FLUSHDB`、`CONFIG`、`DEBUG` 等

## 6. 可扩展性设计

### 6.1 命令扩展

**决策**：设计可扩展的命令处理系统

**理由**：
- **功能扩展**：可以轻松添加新命令
- **定制化**：用户可以根据需要添加自定义命令
- **模块化**：命令处理逻辑可以独立开发和测试

**实现**：
- **CommandHandler 接口**：所有命令处理器的统一接口
- **命令注册机制**：通过注册机制添加新命令
- **命令分发**：根据命令名称自动分发到对应的处理器

### 6.2 存储扩展

**决策**：设计可扩展的存储系统

**理由**：
- **存储选项**：用户可以选择适合自己的存储后端
- **功能定制**：可以根据需要定制存储行为
- **性能优化**：可以针对特定场景优化存储性能

**实现**：
- **MemoryStore 接口**：所有存储实现的统一接口
- **默认实现**：提供基于 Java 集合的默认实现
- **存储配置**：支持通过配置使用自定义存储

### 6.3 插件系统

**决策**：设计插件系统，支持功能扩展

**理由**：
- **功能丰富**：通过插件丰富功能
- **模块化**：插件可以独立开发和部署
- **用户定制**：用户可以根据需要安装插件

**实现**：
- **插件接口**：定义插件的标准接口
- **插件加载**：支持动态加载插件
- **插件管理**：提供插件的注册、激活、停用等功能
- **插件类型**：支持命令插件、存储插件、持久化插件、监控插件等

## 7. 部署设计

### 7.1 多种部署方式

**决策**：支持多种部署方式

**理由**：
- **灵活性**：用户可以根据自己的需求选择合适的部署方式
- **适用场景广**：可以适应不同的应用场景
- **易于集成**：可以轻松集成到现有系统

**部署方式**：
- **独立服务**：作为独立的服务运行
- **嵌入式**：通过 `EmbeddedRedisServer` 嵌入到应用中
- **Docker 容器**：通过 Docker 容器化部署
- **Kubernetes**：在 Kubernetes 集群中部署

### 7.2 配置管理

**决策**：实现灵活的配置管理系统

**理由**：
- **定制化**：用户可以根据需要调整配置
- **环境适应**：可以适应不同的运行环境
- **性能优化**：通过配置优化性能

**实现**：
- **配置文件**：支持通过配置文件设置参数
- **命令行参数**：支持通过命令行参数覆盖配置
- **环境变量**：支持通过环境变量设置配置
- **运行时调整**：部分配置支持运行时调整

## 8. 监控与维护

### 8.1 监控系统

**决策**：实现完善的监控系统

**理由**：
- **可观测性**：提高系统的可观测性
- **问题发现**：及时发现和解决问题
- **性能优化**：通过监控数据优化性能
- **容量规划**：根据监控数据进行容量规划

**实现**：
- **INFO 命令**：返回详细的服务器信息和统计数据
- **脚本统计**：统计 Lua 脚本的执行情况
- **性能指标**：监控命令执行率、连接数、内存使用等指标
- **第三方集成**：支持与 Prometheus、Grafana 等监控系统集成

### 8.2 故障排查

**决策**：提供丰富的故障排查工具

**理由**：
- **问题定位**：快速定位和解决问题
- **系统稳定**：减少故障对系统的影响
- **运维效率**：提高运维人员的工作效率

**实现**：
- **日志系统**：详细的日志记录，包括错误信息、警告等
- **命令统计**：统计命令执行的次数和时间
- **内存分析**：分析内存使用情况，发现内存泄漏
- **网络诊断**：诊断网络连接问题

## 9. 总结

Luban-RDS 的设计决策基于以下核心原则：

1. **兼容性**：与 Redis 完全兼容，支持现有的 Redis 客户端和工具
2. **高性能**：通过多种优化策略，提高系统性能和吞吐量
3. **可靠性**：实现完善的持久化、监控和故障排查机制
4. **安全性**：提供 Lua 沙箱、认证机制等安全特性
5. **可扩展性**：设计插件系统和扩展点，支持功能扩展
6. **易用性**：提供 Spring Boot 集成、多种部署方式等便利特性
7. **可维护性**：模块化设计，清晰的代码结构，便于维护和扩展

这些设计决策共同构成了 Luban-RDS 的技术架构，使其成为一个轻量级、高性能、可靠的 Redis 兼容存储系统。
