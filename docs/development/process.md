# 开发流程

本文档介绍 Luban-RDS 项目的开发流程，帮助开发者了解如何参与项目开发。

## 开发流程概述

1. **需求分析**：理解功能需求和技术要求
2. **代码实现**：编写代码实现功能
3. **测试验证**：运行测试确保功能正确
4. **代码审查**：提交代码进行审查
5. **合并代码**：代码审查通过后合并到主分支

## 分支管理

Luban-RDS 使用 Git 进行版本控制，采用以下分支策略：

- **main**：主分支，包含稳定的代码
- **dev**：开发分支，包含正在开发的功能
- **feature/**：特性分支，用于开发新功能
- **bugfix/**：修复分支，用于修复 bug

## 开发步骤

### 1. 创建分支

从 `dev` 分支创建新的特性分支或修复分支：

```bash
# 创建特性分支
git checkout dev
git pull
git checkout -b feature/your-feature-name

# 或创建修复分支
git checkout dev
git pull
git checkout -b bugfix/your-bugfix-name
```

### 2. 编写代码

- 遵循项目的 [代码规范](standards.md)
- 编写清晰的代码注释
- 确保代码可读性和可维护性
- 实现功能需求

### 3. 运行测试

在提交代码前，运行测试确保功能正确：

```bash
# 运行所有测试
mvn test

# 运行特定模块的测试
mvn test -pl luban-rds-core
```

### 4. 提交代码

- 提交前确保代码通过所有测试
- 编写清晰的提交信息
- 提交信息格式：
  ```
  <类型>: <描述>
  
  <详细说明>
  ```
  类型包括：feat（新功能）、fix（修复）、docs（文档）、style（代码风格）、refactor（重构）、test（测试）、chore（构建/依赖）

```bash
git add .
git commit -m "feat: 添加新功能"
git push origin feature/your-feature-name
```

### 5. 创建 Pull Request

- 在 GitHub 上创建 Pull Request
- 选择 `dev` 作为目标分支
- 填写详细的 Pull Request 描述，包括：
  - 功能描述
  - 实现细节
  - 测试结果
  - 相关问题

### 6. 代码审查

- 等待项目维护者进行代码审查
- 根据审查意见修改代码
- 再次提交修改

### 7. 合并代码

- 代码审查通过后，项目维护者会合并代码到 `dev` 分支
- 定期从 `dev` 分支合并到 `main` 分支，发布新版本

## 开发规范

1. **代码质量**：
   - 遵循项目的代码规范
   - 确保代码可读性和可维护性
   - 避免代码重复

2. **测试**：
   - 为新功能编写单元测试
   - 确保测试覆盖率
   - 运行所有测试确保没有回归

3. **文档**：
   - 更新相关文档
   - 为新功能添加使用示例
   - 确保文档与代码保持一致

4. **提交信息**：
   - 编写清晰、简洁的提交信息
   - 遵循提交信息格式
   - 描述代码变更的目的和影响

## 常见问题

### 代码冲突

当多人同时修改同一文件时，可能会出现代码冲突。解决冲突的步骤：

1. 拉取最新代码：
   ```bash
   git pull
   ```

2. 解决冲突：
   - 编辑冲突文件，手动解决冲突
   - 使用 IDE 的冲突解决工具

3. 提交解决冲突后的代码：
   ```bash
   git add .
   git commit -m "fix: 解决代码冲突"
   git push
   ```

### 测试失败

如果测试失败，需要：

1. 分析测试失败的原因
2. 修复代码问题
3. 重新运行测试确保通过

### 代码审查意见

根据代码审查意见修改代码：

1. 仔细阅读审查意见
2. 理解问题所在
3. 修改代码
4. 再次提交修改

## 发布流程

1. 从 `dev` 分支合并到 `main` 分支
2. 更新版本号
3. 运行构建和测试
4. 发布新版本
5. 更新文档
