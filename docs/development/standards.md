# 代码规范

本文档介绍 Luban-RDS 项目的代码规范，帮助开发者编写一致、清晰、可维护的代码。

## 基本规范

### 缩进
- 使用 4 个空格进行缩进
- 不要使用制表符（Tab）

### 行宽
- 每行代码不超过 120 个字符
- 长行应适当换行，保持代码可读性

### 空格
- 运算符两侧应添加空格：`a = b + c` 而非 `a=b+c`
- 逗号后应添加空格：`func(a, b, c)` 而非 `func(a,b,c)`
- 大括号前应添加空格：`if (condition) {` 而非 `if (condition){`

### 空行
- 方法之间应添加空行
- 逻辑块之间应添加空行
- 文件末尾应添加一个空行

## 命名规范

### 类名
- 使用驼峰命名法（PascalCase）
- 类名应简洁明了，描述类的职责
- 示例：`MemoryStore`、`RedisProtocolParser`

### 方法名
- 使用驼峰命名法（camelCase）
- 方法名应动词开头，描述方法的功能
- 示例：`getValue()`、`setExpiration()`

### 变量名
- 使用驼峰命名法（camelCase）
- 变量名应简洁明了，描述变量的用途
- 示例：`key`、`value`、`expirationTime`

### 常量名
- 使用全大写字母，单词之间用下划线分隔
- 示例：`MAX_KEY_LENGTH`、`DEFAULT_PORT`

### 包名
- 使用小写字母，单词之间用点分隔
- 包名应反映代码的组织结构
- 示例：`com.luban.rds.core`、`com.luban.rds.protocol`

### 枚举名
- 使用驼峰命名法（PascalCase）
- 枚举值使用全大写字母，单词之间用下划线分隔
- 示例：
  ```java
  public enum CommandType {
      GET, SET, DEL
  }
  ```

## 注释规范

### Javadoc 注释
- 为公共类、方法和字段添加 Javadoc 注释
- 注释应包含功能描述、参数说明、返回值说明和异常说明
- 示例：
  ```java
  /**
   * 获取指定键的值
   *
   * @param key 键名
   * @return 键对应的值，如果键不存在则返回 null
   */
  public Object get(String key) {
      // 实现代码
  }
  ```

### 行注释
- 使用 `//` 进行行注释
- 注释应简洁明了，解释代码的逻辑或实现细节
- 示例：
  ```java
  // 检查键是否存在
  if (!keys.contains(key)) {
      return null;
  }
  ```

### 块注释
- 使用 `/* */` 进行块注释
- 块注释用于解释复杂的代码逻辑或算法
- 示例：
  ```java
  /*
   * 实现 LRU 缓存淘汰策略
   * 1. 当缓存容量达到上限时，移除最近最少使用的键
   * 2. 更新访问时间戳
   */
  private void evict() {
      // 实现代码
  }
  ```

### 中文注释
- 对于团队内部的代码，注释可以使用中文
- 确保注释清晰、准确，与代码保持一致

## 代码结构规范

### 类结构
- 类的成员变量应放在类的顶部
- 构造方法应放在成员变量之后
- 普通方法应按功能分组，放在构造方法之后
- 静态方法和静态变量应放在类的底部

### 方法结构
- 方法应保持简洁，每个方法只负责一个功能
- 方法长度不宜过长，一般不超过 50 行
- 复杂的方法应拆分为多个小方法

### 异常处理
- 应使用 try-catch 块处理异常
- 异常处理应具体，避免捕获通用的 Exception
- 异常信息应清晰，说明异常的原因
- 示例：
  ```java
  try {
      // 可能抛出异常的代码
  } catch (IOException e) {
      logger.error("IO 异常: {}", e.getMessage());
      throw new RedisException("操作失败: " + e.getMessage());
  }
  ```

## 编程实践规范

### 代码可读性
- 代码应易于理解，避免使用复杂的表达式
- 应使用有意义的变量名和方法名
- 应添加适当的注释解释复杂的逻辑

### 代码可维护性
- 应避免代码重复，使用方法或工具类提取重复代码
- 应使用设计模式提高代码的可维护性
- 应遵循 SOLID 原则

### 性能优化
- 应注意代码的性能，避免不必要的计算和操作
- 应使用合适的数据结构和算法
- 应避免在循环中创建对象

### 安全性
- 应注意代码的安全性，避免安全漏洞
- 应验证用户输入，防止注入攻击
- 应使用安全的密码存储方式

## 测试规范

### 单元测试
- 应为所有公共方法编写单元测试
- 测试应覆盖正常情况和异常情况
- 测试方法名应清晰，描述测试的内容
- 示例：
  ```java
  @Test
  public void testGetExistingKey() {
      // 测试获取存在的键
      memoryStore.set("key", "value");
      assertEquals("value", memoryStore.get("key"));
  }
  ```

### 集成测试
- 应为核心功能编写集成测试
- 测试应模拟真实的使用场景
- 测试应验证组件之间的交互

## 工具和配置

### IDE 配置
- 推荐使用 IntelliJ IDEA 或 Eclipse
- 应配置 IDE 使用 4 个空格进行缩进
- 应配置 IDE 自动格式化代码

### 构建工具
- 使用 Maven 进行项目构建
- 应在 pom.xml 中配置编译插件，确保代码符合规范

### 代码质量工具
- 推荐使用 SonarQube 进行代码质量检查
- 应定期运行代码质量检查，确保代码质量

## 常见问题

### 命名不一致
- 问题：变量名、方法名或类名命名不一致
- 解决：遵循命名规范，使用有意义的名称

### 代码过长
- 问题：方法或类过长，难以理解和维护
- 解决：拆分方法，提取重复代码，使用设计模式

### 注释缺失
- 问题：代码缺少注释，难以理解
- 解决：为公共方法和复杂逻辑添加注释

### 异常处理不当
- 问题：异常处理不当，可能导致程序崩溃或安全漏洞
- 解决：使用合适的异常处理方式，捕获具体的异常类型

### 测试覆盖不足
- 问题：测试覆盖不足，可能导致代码中的 bug 未被发现
- 解决：为所有公共方法编写单元测试，覆盖正常情况和异常情况
